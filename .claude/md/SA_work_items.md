## 一、身分認證與授權機制

### 1-1 LINE@ Token 驗證流程

- [ ] 預約頁面入口管制：僅接受帶有 Token 的請求（拒絕未帶 Token 的直接訪問）
- [ ] 後端攔截 LINE userId：從 Token 連結中抓取使用者 LINE userId
- [ ] 已綁定用戶自動登入：查詢 DB 有對應 LINE userId 則核發 Session / JWT，免登入進入
- [ ] 新用戶強制導向註冊頁：查無 LINE userId 時跳轉至信箱+密碼註冊頁
- [ ] 註冊 API 背景自動綁定 LINE userId

### 1-2 Web 端常規登入

- [ ] 信箱 + 密碼登入 API
- [ ] 密碼重設功能（發送含驗證連結的 Email）

### 1-3 LINE@ 好友防呆機制

- [ ] 「請先加 LINE@」引導頁面實作
- [ ] 進入預約核心前，檢查是否已完成 LINE@ 好友綁定

## 二、即時預約引擎

### 2-1 可用時段動態運算 API（Availability API）

- [ ] 實作公式：可用時段 = 美容師班表 - 已成立預約 - Buffer time - 休假日
- [ ] 回傳僅包含「真正可預約」空檔的時段陣列
- [ ] 預約窗口限制：僅回傳未來 N 天內時段（N 為系統參數，預設 30 天）

### 2-2 交易鎖定與防衝突（Concurrency Control）

- [ ] 客戶送出預約時，DB 層即時鎖定時段（Redis 或 DB Transaction）
- [ ] 禁止同一美容師同一時間重複預約
- [ ] 強制執行療程間隔（Buffer time）

### 2-3 業務規則驗證（Business Rules Validation）

- [ ] 預約窗口天數設定為可動態調整的系統參數
- [ ] 跨店排班限制：同一美容師單日僅能在單一據點服務
- [ ] 選擇據點後，僅顯示「該地區會做此療程的美容師」

### 2-4 狀態異動限制

- [ ] 取消 API：預約時間 24 小時前才允許自助取消
- [ ] 24 小時內取消：阻擋並回傳錯誤碼，前端提示「需聯繫總部」

## 補充：系統參數彙整（需統一管理）

| 參數名稱                | 說明                 | 預設值     |
| ----------------------- | -------------------- | ---------- |
| `booking_window_days`   | 預約窗口天數         | 30 天      |
| `buffer_time_minutes`   | 療程間隔時間         | 依療程設定 |
| `cancel_deadline_hours` | 自助取消截止時數     | 24 小時    |
| `line_menu_token`       | LINE@ 圖文選單 Token | 後台設定   |
| `line_push_templates`   | LINE 推播模板        | 後台設定   |
